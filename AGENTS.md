# AGENTS.md

This tree is the SVR4 kernel source used by the bootstrap build. It is an
imported/checked-out source tree, not the top-level bootstrap recipe repo.

## Build Context

- The kernel build is driven by `build.py` and YAML specs under
  `build-specs/i386/kernel/`.
- Normal build artifacts live outside this tree, usually under
  `../../svr4-bootstrap-build/build/uts/i386/` or some other xbstrap build location 
  when invoked from the top-level bootstrap checkout.
- Low-level machine-layer objects are generated by `tools/uts_ml_build.py`. In
  particular, `locore.s` is assembled from several `i386/ml/*.s` inputs after a
  preprocessing and normalization pass.
- The build is 32-bit i386. Do not introduce SSE/MMX/XSAVE assumptions into the
  kernel code itself - handling support specifically at the request of the user
  may be permissible, but the kernel itself must still be able to run on a 386,
  if the correct toolchain/build arguments are used.
- Aside from the note above, the default toolchain used by the kernel and the
  bootstrap tree builds for a i686 (Pentium Pro) baseline.

## Kernel Source Style

- Much of the original source is K&R C. Match the local file style for small
  edits.
- New files may use ANSI C. Substantial rewrites of a function or file may also
  modernize that rewritten scope to ANSI C.
- Keep changes narrowly scoped. This code still aims to preserve original SVR4
  behavior where possible.

## Feature Policy

- Ask before adding or removing kernel features, syscall behavior, ABI-visible
  fields, or driver behavior.
- Bug fixes are welcome.
- Pure additions that preserve existing behavior are usually preferable to
  userland workarounds, but still consider compatibility with old SVR4 software.
- Avoid changing existing syscall semantics unless explicitly approved.

## Kernel Defines

- Quite a few paths in the kernel are gated on various defines; read the build-
  spec files to be sure about which ones are enabled where.
- This is a living list, with the intention of slowly creating a bit of
  documentation on what defines mean exactly what, and what consequences there
  are of them. If you have to research one in-depth, add a entry for it here,
  but don't add any information about any specific issues you had. Just add a
  brief description about them.
- `VPIX` is VP/ix virtual-8086/DOS support. It is not expected to be enabled in
  the current build.
- `MERGE386` is Locus Merge/386 DOS emulation support and is also not expected
  to be enabled.

## Testing

- The kernel package is generally build as a xbstrap package, and is thus normally
  checked out in the bootstrap tree.
- If relevant, and you need to test stuff here, read the `../AGENTS.md` file, if it
  exists. If it does not, and the user hasn't told you anything else in regards to
  xbstrap, assume you can't test the code automatically, and ask the user to test
  changes.
